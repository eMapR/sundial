# Sundial
Workflow / framework for deep learning with data sourced from Google Earth Engine on a bare metal machine.

## Introduction

Google Earth Engine provides an API endpoint which allows you to quickly download images up to 48 MB, give or take a few MB. This is relatively small as far as spatial science goes, but for machine learning, it is an ideal tool for getting a large number of training chips for deep learning research. GEE also provides easy access to a plethora of pre-processsed image collections.

This repo contains scripts to download square chips via polygons sourced from shapefiles and run them through a neural network of your own choosing. Although Google already provides powerful tools for deep learning integrated with GEE, it can be beneficial and/or cost-effective to perform analysis on your own GPU(s). The intended use is time series analysis on an HPC however, the scripts will work on any machine with Nvidia GPU(s).

## Basic Usage

```console
make

Welcome to Sundial!

          Current config directory: configs/$(SUNDIAL_EXPERIMENT_BASE_NAME)

    Methods:

Setup methods for starting experiment.

        setup_env:      Sets up Sundial directories and environment. The existing environment will be overwritten.
        setup:          Sets up Sundial experiment directories and config files using defaults found in **sundial/settings.py.**

Sampling methods for generating image chips and indexing them.

        download:       Downloads image chips for experiment using sample polygons generated by sample method.
        annotate:       Collects image annotations for experiment using sample polygons generated by sample method.
        stats:          Creates indicies for training from chip and anno data using indexer in configs/pre_/pipeline.yaml.

Training methods. There should be base yaml with model information at configs/pre_/base.yaml

        fit:            Train model using configs/pre_/fit.yaml. This will run a full optimization routine.
        validate:       Validate model using configs/pre_/validate.yaml. This will run a single epoch.
        test:           Test model using configs/pre_/test.yaml. This will run a single epoch.
        predict:        Predict an image using configs/pre_/predict.yaml. This will run a single epoch.
        package:        Convert predictions in predictions/pre_ to GeoTIFF and compress for download.

Debug / help methods

        status:         Check status of all jobs for user.
        vars:           Print all Sundial variables.
        (method)_err:   Print ERRORs and CRITICALs in log file and print stderr from file on HPC.

Cleaning methods

        clean_expt:     Removes all logs, checkpoints, and predictions for experiment.
        clean_dnld:     Removes all imagery data for experiment.
        clean_anno:     Removes all annotation data for experiment.
        clean_ckpt:     Removes all checkpoints for experiment.
        clean_pred:     Removes all predictions for experiment.
        clean_nuke:     Removes all data for experiment including configs.

    Variables:
    These may be set as submake flags or as environment variables.

        SUNDIAL_BASE_PATH:           Base path for Sundial scripts. Default: 'shell pwd' of makefile
        SUNDIAL_SHAPE_NAME:          Shapefile name. REQUIRED
        SUNDIAL_EXPERIMENT_PREFIX:   Sundial experiment name prefix used for experiment naming. REQUIRED
        SUNDIAL_EXPERIMENT_SUFFIX:   Experiment suffix used only for logging.
        SUNDIAL_ENV_NAME:            Sundial environment name. Default: 'sundial'
        SUNDIAL_PROCESSING:          Sundial processing method. Default: 'hpc'
        SUNDIAL_NODELIST:            Node within hpc to run job. Default: any node
        SUNDIAL_GPU_PARTITION:       Partition within hpc to run gpu based jobs.
        SUNDIAL_CPU_PARTITION:       Partition within hpc to run cpu based jobs.
        SUNDIAL_PACKAGE_FORMAT:      Format to package predictions. Default: 'tar'

```

### 1. Setup directories and conda environment.
See [Google Earth Documentation](https://developers.google.com/earth-engine/guides/auth) for mode details on authentication.
```console
# Clone repository cd into it. Note: Pretrained models should be cloned from their respective repos into sundial/models/backbones, however, you can just as well load in as a plain lightning module. (eg Prithvi in the the subdir 'prithvi')
git clone https://github.com/eMapR/sundial.git
cd sundial

# Create environment in one of two ways
pip install -r requirements_x86.txt

# Authenticate on Google Earth Engine.
earthengine authenticate
```

### 2. Generate experiment directories and config files for running submakes.
Experiment directories will be generated with the appropriate paths for all files that need to be created and read per experiment. Default configs found in **sundial/settings.py** and **sundial/pipeline/settings.py** will be copied to the appropriate config location and any edits to them will automatically supersede both pipeline/settings.py and settings.py values. Config files must be created before any other experiment submake.
```console
# Set environment variables for experiment.
# you can also set them in your bash terminal
    export SUNDIAL_PROCESSING=local \ # only necessary if you want to run on local bare metal
    export SUNDIAL_SAMPLE_NAME=shapefile_name \
    export SUNDIAL_EXPERIMENT_PREFIX=5c # experiment name 'prefix'
make setup
```

### 3. Generate annotation images from original shapefile and sample metadata.
Features in the shapefile are used to create annotations by pixel using the **PIPELINE_CONFIG.class_columns** setting. The class columns are used to delineate between classes so appropriate raster masks are created according to traditional deep learning image labels. The annotation image tensor shape is (N C T H W). This step is only necessary for supervised learning. Note: Annotations are saved as their own image. This may not be as storage efficient but makes loading into Pytorch much simpler and quicker. The config file to edit is **config/$(SUNDIAL_EXPERIMENT_PREFIX)_$(SUNDIAL_SAMPLE_NAME)/sample.yaml**. See **sundial/pipeline/settings.py** for more information.
```console
make annotate
```

### 4. Downloading using the generated chip samples via GEE.
GEE is used to generate raster data based on polygons from the shapefile given in the sample. Download limits apply so use your own discretion. I found the limit to be chips of shape (256, 256, 256) to be the upper limit but this may vary depending on processing, scale, projection, etc. Depending on your internet speed, ~32k chips of shape (32, 256, 256) can be downloaded in just a few hours. This is usually more than enough data. Rasters are saved to **samples/$(SUNDIAL_EXPERIMENT_PREFIX)_$(SUNDIAL_SAMPLE_NAME)** using the specified file format. Both download and annotate will default to zarr files using xarray but other file types can be specified. The config file to edit is **config/$(SUNDIAL_EXPERIMENT_PREFIX)_$(SUNDIAL_SAMPLE_NAME)/sample.yaml**. See **sundial/pipeline/settings.py** for more information. The download will use a ee factory from 'sundial/pipeline/ee_factories.py'
```console
make download
```

### 5. Train a model using the downloaded chips.
Included in this repo is a simple segmentation model using a fully convolutional network built on [Prithvi's foundation model](https://huggingface.co/ibm-nasa-geospatial/Prithvi-100M) that needs to be cloned to models/backbones and finetuned. Using the CLI, you can mix and match models. See [Pytorch Lightning Docs](https://lightning.ai/docs/pytorch/stable/cli/lightning_cli.html) for more details on config files. For now the runner will import models by name from the models module using a provided class_path and init_args. Write additional models there. Any paths to chip or annotation data created in the previous steps are automatically loaded. A checkpoint module is also by default included as callbacks. The config file to edit is **config/$(SUNDIAL_EXPERIMENT_PREFIX)_$(SUNDIAL_SAMPLE_NAME)/fit.yaml**.  See **sundial/settings.py** for more information.
```console
make fit
```

### 6. Test Model.
A separate test workflow can be run using a similar format as the fit.yaml. The config file to edit is **config/$(SUNDIAL_EXPERIMENT_PREFIX)_$(SUNDIAL_SAMPLE_NAME)/test.yaml**. In addition to loading a separate dataset and some eval functions of lightning, there is some autoloading of best checkpoints. See sundial/runner.py for more information.
```console
make test
```

## Config files
While the framework for deep learning is build on [Pytorch Lightning](https://lightning.ai/docs/pytorch/stable/cli/lightning_cli.html), the pipeline down from GEE can be configured using a similar format. See sundial/pipeline/settings.py for more details and configs directory for examples. In addition, images are generated using an annual medoid composite by default since for now, that is our focus, but another image factory function can be provided to the class if you define it in pipeline/ee_image_factories.py. See sundial/pipeline/settings.py for more information.
